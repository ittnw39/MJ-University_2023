요구사항 정리

requirement specific

요구사항 수집해서 정리 -> 분석 -> 설계 -> 구현 -> 테스트

ㄴ 일반적인 소프트웨어를 개발하는 행위를 구성함. - 액티비티

사용자 요구사항 정의  -> 사용자가 처음에 있다. 정보 처리 시스템(사람의 두뇌가 해야할 일들을 처리하는 것이다.)

업무가 대신하는 사람의 행위 1.물리적 행위 2.지적 행위 -> 사람이 만든 가상의 시스템에서 업무를 처리할 때 정보 처리 시스템이라고 한다.

정보 처리 시스템은 고객이 존재함. actor
요구사항은 현실에서 제한된 공간(은행, 컨텍스트)과 사용자 문맥(고객의 요구사항). 공간에서 사용자에게 무언가를 제공해주는 것.
사용자의 요구사항은 시스템의 경계를 정리함으로서 나타난다.

요구사항에 대한 것들을 정의할 때 시나리오가 사용자와 시스템의 상호작용에 관한 것들이 어떤지를 쓴다.

시스템안에 도메인 엑스퍼트(업무 전문가, 업무가 어떻게 처리되는지 아는 사람, 해당 영역의 전문가)- 사용자에게 서비스를 제공하기위해 있다.

항상 두 종류의 사람이 있다. 사용자는 요구만하고 도메인 전문가는 어떻게 할 것인지를 담당한다.

사용자 시나리오---

- 수강신청 = 동사, 기능(사용자가 원하는 것, function, 함수, 어떤 집합의 각 원소를 다른 어떤 집합의 원소에 대응시키는 규칙)
세상을 표현하는 것 중 가장 일반적인 것이 자연어. 이를 학문으로 나눠놓음. 복잡해서 세상의 규칙을 특정 관점에서 뜯어놓음.
함수의 규칙을 정하는 것 = 알고리즘

자연어는 우리가 세상을 표현하는 방법. 언어에는 세상이 들어가 있어야 함. 자연어가 보는 세상은 명사와 동사, 부사....로 이루어져 있다. 품사는 단어가 가진 역할
우주는 3차원 공간과 시간. 명사는 시간이 지나도 변하지 않는 구조이다. - 기준으로 삼을 수 있다. 
동사는 대상의 구조가 가진 내부의 값이 변하는 것이다. 

서비스는 사용자가 지불하고 원하는 것을 받는 것 = 기능

함수 멀티 스택 = 알고리즘

결과가 모호하지 않다 = 확정적이지 않다.
하나의 문장이 두개가 대응되면 모호하다고 한다. 결과를 쌍하서 복잡도를 올린다.

함수 다음은 관계

n : n = 관계

함수, 알고리즘..

함수는 입력, 출력이 있다. 입력의 규칙 - 알고리즘을 통해서 출력이 나온다.

아웃풋 - 강좌 리스트->수강 신청(리스트에 강좌리스트를 추가, 함수의 입력)->수강신청 리스트(수강신청 함수(강좌리스트가 파라미터로 들어감.))

데이터 플로우 다이어그램 DFD - 중요

함수는 데이터가 변형되어 흘러가는 곳으로 본다.
알고리즘을 표현하는 방법 중 하나.

데이터 플로우 = 함수, 기능, 알고리즘

동사도 기능이다.. 명사에 따라(규칙에 따라) 결과가 달라짐. = 명사에 따라서 행위가 달라짐.

어떤 기능이 필요한지 뽑는다. 명사의 흐름을 정해야 한다. 해당 영역 지식을 참조해서 기능의 흐름을 서술할 수 있어야 함.
